# 공연 검색 기능 설계서 (Search System Design)

| 버전 | 날짜 | 작성자 | 변경 내용 |
|------|------|--------|----------|
| 1.0  | 2026-03-07 | Gemini | 초기 설계 및 분석 내용 작성 |

---

## 1. 개요
사용자가 원하는 공연을 쉽고 빠르게 찾을 수 있도록 통합 검색, 자동완성, 필터링 및 정렬 기능을 제공합니다.

---

## 2. 요구사항 상세

### 2.1 검색 대상
*   **통합 검색 범위**: 공연 제목, 공연 카테고리(장르), 공연장 주소
*   **검색 대상 상태**:
    *   **자동완성**: 판매 예정, 판매 중인 공연만 노출 (최대 6개)
    *   **검색 목록**: 판매 종료를 포함한 모든 공연 대상 (필터 적용 가능)

### 2.2 검색창 자동완성 (Autocomplete)
*   **노출 항목**: 공연 썸네일, 제목, 카테고리, 지역(행정구역 17개)
*   **동작**: 검색어 입력 시 실시간으로 최대 6개의 관련 상품 노출
*   **지역 표기**: [썸네일] 2026 권진아 콘서트 (콘서트, 경남) 형식

### 2.3 검색 목록 (Search List)
*   **UI/UX**: 인피니티 스크롤 적용
*   **필터링**:
    *   **판매 상태**: 판매 예정 / 판매 중 / 판매 종료
    *   **장르**: 카테고리 기반 필터링
    *   **지역**: 행정구역 17개 기반 필터링 (공연장 주소 매핑)
*   **실시간 집계**: 필터 선택 시 각 조건에 해당하는 상품 수(Count)를 실시간으로 표시
*   **정렬 조건**:
    *   **예매 많은 순**: 성공한 예매 건수 기준 내림차순
    *   **종료 임박 순**: 공연 종료일(`endDate`) 오름차순
    *   **최근 등록 순**: 생성 일시(`createdAt`) 내림차순

## 3. 기술적 분석 및 설계

### 3.1 공연 상태 정의 로직
공연의 판매 상태는 다음 필드들을 조합하여 런타임에 계산합니다.
*   **판매 예정**: `now() < saleStartDateTime`
*   **판매 중**: `saleStartDateTime <= now() <= endDate`
*   **판매 종료**: `endDate < now()`

### 3.2 행정구역(17개) 매핑 전략
홈 화면 큐레이션용인 `HomeSectionTag`와 검색/필터용 지역 정보를 분리하여 설계합니다.
*   **독립적인 `Region` Enum 도입**: 17개 광역자치단체(서울, 인천, 경기, 강원, 충북, 충남, 대전, 세종, 전북, 전남, 광주, 경북, 경남, 대구, 울산, 부산, 제주)를 정의하는 별도의 열거형을 생성합니다.
*   **매핑 로직**: `RegionMapper` 또는 별도의 유틸리티를 통해 공연장 주소(`Venue.address`)에서 `Region`을 추출합니다.
*   **분리 운영**: 
    *   **홈 화면**: 기존 `HomeSectionTag`의 주요 6개 지역 섹션을 유지하여 관리 효율성을 높입니다.
    *   **검색/필터**: 17개 행정구역 전체를 대상으로 필터링 및 실시간 개수 집계를 수행합니다.

### 3.3 예매 수량 집계 (정렬용)
*   `Performance` - `PerformanceSchedule` - `Booking` 관계를 Join하여 `BookingStatus.CONFIRMED` 상태의 건수를 집계합니다.
*   성능 최적화를 위해 필요 시 `Performance` 엔티티에 `totalBookingCount` 컬럼을 추가하고 결제 완료 시점에 업데이트하는 비정규화 방식을 검토할 수 있습니다.

### 3.4 API 설계

#### 1) 검색 자동완성
*   **Endpoint**: `GET /api/performances/search/autocomplete`
*   **Query Params**: `keyword` (검색어)
*   **Response**: `List<PerformanceAutocompleteResponse>`

#### 2) 검색 목록 조회
*   **Endpoint**: `GET /api/performances/search`
*   **Query Params**:
    *   `keyword`: 검색어
    *   `status`: 판매상태 필터
    *   `category`: 장르 필터
    *   `regionTag`: 지역 필터
    *   `sort`: 정렬 방식
    *   `page`, `size`: 페이징 정보
*   **Response**: `PerformanceSearchListResponse` (목록 데이터 + 필터별 카운트 메타 정보)

## 4. 구현 고려사항
1.  **동적 쿼리 처리**: Querydsl 미도입 상태이므로 `JpaSpecificationExecutor` 또는 `EntityManager`를 활용한 커스텀 Repository 작성이 필요합니다.
2.  **성능**: 인피니티 스크롤 대응을 위해 `Slice` 또는 `Page` 처리를 최적화하고, 대량 데이터 검색 시 인덱스 전략을 수립해야 합니다.
3.  **데이터 정합성**: 지역 태그 자동 매핑 시 신규 17개 구역이 정확히 반영되도록 기존 데이터 재정비가 필요할 수 있습니다.
